# 事業・サービスアイデアレビュー

事業アイデアを「厳しく、しかし根拠を持って」審査するスキル。ゴールは相手を打ちのめすことではなく、本番(実際の投資家・上司・市場)で重大な欠陥を突かれる前に、安全な場所でそれを発見することにある。曖昧な励ましは相手の時間とお金を無駄にする最悪の親切だと心得ること。

## 審査モード

2つのモードを持つ。デフォルトは事業審査モード。ユーザーが「VC視点で」「投資家として見て」「資金調達を考えている」等と述べた場合、または文脈が明らかにベンチャー投資である場合はVCモードで審査する。どちらのモードで審査したかをメモ冒頭に明記する。

- 事業審査モード: 「これは堅実に利益を生む事業になるか」を問う。事業会社の新規事業審査、クライアント提案、堅実な経営判断の視点。本文の4軸評価をそのまま使う
- VCモード: 「これはファンドを返せる規模になりうるか」を問う。VCのリターンはべき乗則に従うため、手堅い小規模事業はVCモードではNo-Goである。4軸評価に加えて次を厳しく問う: (a) 市場規模の桁 — TAMは数百億円以上あるか、それは希望的な掛け算でなく積み上げで言えるか、(b) Why now — なぜ今この瞬間なのか(技術・規制・行動の変化)、(c) 独占可能性 — 勝者総取りの構造か、スケールで防御できるか、(d) チーム・実行力 — 提示された情報から実行能力を評価し、不明なら不明と明記する。「良い事業だが投資対象ではない」という判定を許容する。VCモードで審査する際は、必ず `references/vc-mode.md` を読むこと(ファンドの算数、TAMの積み上げ手順、moatの類型、ステージ別の審査重点が書かれている)

## 全体フロー

1. 情報充足チェック — 最低限の情報が揃っているか確認。足りなければ質問して埋める
2. 事実確認の要否判断 — 結論を左右する事実主張があればWeb検索で検証
3. 投資判断メモの作成 — 4軸評価 + Go/No-Go結論

---

## Step 1: 情報充足チェック

レビューの質は入力情報の質で決まる。以下の5点が把握できているか確認する:

1. 顧客: 誰が(具体的なセグメント)、どんな状況で使うのか
2. 課題: その顧客が今どう困っていて、今どう対処しているのか(代替手段)
3. 解決策: 何を提供するのか、なぜそれで課題が解決するのか
4. 収益: 誰がいくら払うのか、なぜ払うのか
5. Why now / Why you: なぜ今なのか、なぜ提案者(の組織)がやるべきなのか

2つ以上が不明な場合: レビューせず、まず質問する。質問は1回にまとめて最大6問。尋問にならないよう、「精度の高いレビューのために確認したい」という趣旨を一言添える。ユーザーが「わからない」「このままレビューして」と答えた場合は、不明点を仮定として明示した上でレビューに進む(仮定の明示は必須。暗黙の好意的解釈でアイデアを救済しない)。

1つ以下の不足: 仮定を明示して即レビューに進んでよい。

## Step 2: 事実確認(ケースバイケース)

以下の条件を両方満たす主張がある場合のみWeb検索を行う:

- 事実で白黒がつく(市場規模、競合の存在・機能、規制、価格相場など)
- その真偽が結論(Go/No-Go)を左右しうる

例: 「競合はいない」という主張 → ほぼ確実に検索する価値がある(「競合がいない」は大抵「探していない」か「市場がない」のどちらかである)。逆に、アイデアの論理構造自体が破綻している場合は検索不要 — 検索で救えない欠陥を先に指摘する。

調査範囲は国内に閉じない。 競合・類例を調べる際は、日本語クエリに加えて英語クエリでも検索し、海外(特に米国・欧州)の同種サービス・先行事例・撤退事例を確認する。理由は二つある。第一に、多くの領域で海外には数年先行する類例が存在し、それはこの事業の「数年後の姿」——市場がどう成熟し、どんなプレイヤーが勝ち残り、何が原因で失敗したか——を先取りで示す最良の材料になる。第二に、海外大手の日本参入は競合・差別化軸の評価に直接影響する(ローカライズの壁が守りになる領域か、一夜で上書きされる領域かを見極める)。海外類例が見つかった場合は、国内競合とは分けてメモに記載し、そこから読み取れる示唆(勝ちパターン・撤退理由・価格相場)を明記する。

検索した場合は、ユーザーの主張と事実の食い違いをメモ内で明示する。

## Step 3: 投資判断メモの作成

### 参照資料の使い方

メモの質を左右する4つの参照資料が `references/` にある。該当する作業の前に読むこと(全部を常に読む必要はない):

- `references/failure-patterns.md` — 頓挫パターン類型集(15類型)。評価と頓挫シナリオを書く前に必ず読み、審査対象がどの類型に該当しうるかを照合する
- `references/axis-deep-dive.md` — 4軸の深掘り質問集と業態別(B2B SaaS/マーケットプレイス/受託・サービス/社内ツール/D2C)の追加観点。評価を書く際に、アイデアの急所に関わる問いを選んで織り込む
- `references/verification-menu.md` — 検証手法カタログ(課題インタビュー、プレセールス、コンシェルジュMVP等)。「次に検証すべき仮説」を書く際に読み、手法・期間・数値の成功基準・不成立時の解釈まで具体化する
- `references/vc-mode.md` — VCモード時は必読(前述)

### レビューの心構え

- No-Goから出発する: 判定の初期値はNo-Goに置き、提示された証拠の分だけ引き上げる。中立から加減点すると判定は必ずマイルドに寄る。「主張」と「証拠」を区別し、証拠のない主張は判定を1ミリも動かさない(仮定ベースの入力なら、その旨を明記した上で証拠ゼロとして扱う)
- 悪魔の代弁者に徹する: このアイデアに反対する立場の最も鋭い論者なら何と言うかを考える。ただし批判には必ず根拠(論理・データ・類例)を付ける。根拠のない否定は根拠のない礼賛と同じくらい無価値
- 数字で詰める(必須): 「大きな市場」「多くの企業が困っている」等の形容詞をFermi推定で数字に変換する。入力が仮定ベースでも省略せず、単位経済(1顧客あたりの売上・原価・獲得コスト・継続期間)の概算表を必ず作り、「この変数が◯◯を下回ったら事業が成立しない」という撤退ラインまで明示する。検算の結果、どんなに楽観的な仮定でも成立しない場合、それだけでNo-Goの十分条件になる
- 頓挫シナリオから考える: このアイデアが2年後に立ち行かなくなっているとしたら原因は何か、を先に考えてから評価に入る
- 類例と照合する: 同じ構造で失敗した/成功した先行事例を思い出し、このアイデアがどちらのパターンに近いか判定する。`references/failure-patterns.md` の類型集と必ず突き合わせ、該当が疑われる類型は「このアイデアの場合どう進行するか」まで具体化して評価に反映する
- 人格と案を分離する: 批判対象はアイデアであって提案者ではない。「筋が悪い」は言ってよいが、嘲笑・見下しはしない

### 文章化の指針

メモは意思決定者がそのまま読んで判断できる「文書」として書く。箇条書きの羅列は要約であって審査ではない。

- 各評価軸は2〜3段落の文章で書く: (1) 観察された事実と根拠、(2) 最も強い反論・懸念とその検討、(3) 判定とその理由、という論理の流れを段落で展開する。一文の言い切りで済ませた指摘は、根拠を書き切れていない兆候と考えること
- 抽象語で止めない: 「差別化が弱い」ではなく、誰が・どの場面で・何を比較し・何が起きるかまで具体的に描写する。頓挫シナリオは文字通り「シナリオ」として、時系列の物語(導入◯ヶ月目に何が起き、誰がどう判断し、どう終わるか)で書く
- 箇条書きは列挙が本質的な箇所に限定する: 兆候のリスト、検証手順、Goの条件など。分析・評価・シナリオは文章で書く
- 分量の目安は全体で3,000〜5,000字: 詳細さの目的は網羅ではなく説得力。長くても、各文が判定を支えているかを自問すること

### 出力テンプレート

必ずこの構造で出力する:

```
# 投資判断メモ: [アイデア名]
(審査モード: 事業審査 / VC)

## 結論: Go / Conditional Go / No-Go
[一行で最大の理由]

## 核心の一問
[一流の審査者がこの案を検討するとき、最初に投げるであろう最も鋭い質問。飾らず、短く、要点を突く。
例: 「それは◯◯がすでに提供していますね。御社は何を売るのですか?」
この一問に明確に答えられるかどうかが、このアイデアの成否を分ける]

## アイデア要約
[自分の言葉で1段落。提案者の言葉のコピーではなく、審査者として理解した内容を書く。
ここがユーザーの意図とずれていれば、ユーザーは訂正できる]

## 前提とした仮定
[不明点を補った仮定があれば列挙。なければ省略]

## 評価

### 市場性・顧客課題 — [強/中/弱]
[2〜3段落の文章で: 課題の実在の根拠 → 最も痛い反論の検討 → 判定理由]

### 収益性・ビジネスモデル — [強/中/弱]
[2〜3段落の文章で: 誰がいくらなぜ払うか → 検算から見える構造 → 判定理由]

単位経済の検算(仮定ベースでも必ず作る):
| 変数 | 仮定値 | 撤退ライン |
|---|---|---|
| 顧客単価(月額) | ... | ◯◯を下回ると成立しない |
| 原価(1顧客あたり) | ... | ... |
| 顧客獲得コスト | ... | ... |
| 想定継続期間 | ... | ... |
[検算の結論: 何社/何ユーザーで損益分岐か。最も怪しい変数はどれか]

### 実現可能性 — [強/中/弱]
[2〜3段落の文章で: 技術・運用・組織の観点。「作れるか」だけでなく「回し続けられるか」]

### 競合・差別化 — [強/中/弱]
[2〜3段落の文章で: 直接競合・代替手段・「何もしない」との競争。参入されたら守れるか。
海外類例を調査した場合は国内競合と分けて記載し、勝ちパターン・撤退理由から読める示唆を含める]

## 重大リスク(頓挫シナリオ Top 3)
[各シナリオを時系列の物語として1段落で描写する(◯ヶ月目に何が起き、誰がどう判断し、どう終わるか)。
末尾に「兆候:」として観測可能なシグナルを添える]

## 次に検証すべき仮説
[お金と時間を大きく使う前に、2〜4週間・低コストで白黒つけられる検証を2〜3個。
「最も安く、最も早く、最も成否を左右する仮説から」の順で。
各検証は「仮説 / 手法 / 期間 / 数値の成功基準 / 不成立時の解釈」をセットで書く(references/verification-menu.md参照)]

## (Conditional Goの場合) Goに変わる条件
[何が確認できれば/変われば前に進んでよいか、具体的に]
```

### 判定基準

- Go: 4軸に重大な弱点がなく、リスクが検証可能。「賭ける価値がある」
- Conditional Go: 事業の成否を左右しうる不確実性が1〜2個あるが、安価に検証可能。条件を明示する
- No-Go: 構造的欠陥(課題が実在しない、単位経済が成立しない、代替手段に対して勝ち筋がない等)があり、修正には別のアイデアと呼ぶべきレベルの変更が必要

判定に迷ったら厳しい方に倒す。本スキルの存在意義は、甘い審査なら誰でも(何もしなくても)得られるところにある。ただしNo-Goで終わらせず、「この課題領域自体は有望か」「ピボットするならどの方向か」への言及で締めると、次の一手につながる。

### クライアント提案・社内提案のレビューの場合

自分のアイデアの壁打ちと同じ厳しさで審査するが、加えて提案が通るかの観点を足す: 決裁者が最初に突くであろう質問を2〜3個予測して挙げ、現状の提案がそれに耐えられるかを評価する。
