# 検証手法カタログ

「次に検証すべき仮説」セクションを書く際に読む。仮説に対して適切な手法を選び、期間・費用・成功基準まで具体化して提案するための道具箱。

## 手法選択の原則

1. 仮説を先に、手法を後に: 「何が真なら前進してよいか」を文にしてから手法を選ぶ。手法ありきの検証(とりあえずアンケート等)は何も証明しない
2. もっとも成否を左右する仮説から: 全仮説が真でないと成立しない事業なら、最も疑わしい仮説を最初に潰すのが最速である
3. 成功基準は事前に数値で: 「好評だったら前進」は検証ではない。「10人中◯人が◯◯したら真とみなす」を検証開始前に固定する
4. 言葉より行動を証拠とする: 「買いたい」という発言は証拠として弱い。時間を割く、データを預ける、前払いする、稟議を起案する——顧客のコストを伴う行動だけが強い証拠である

## 手法一覧

### 課題インタビュー
- 検証できる仮説: 課題の実在、深刻度、現在の代替手段
- 期間/費用: 1〜3週間 / ほぼゼロ(対象者への謝礼程度)
- やり方の要点: 製品の話をせず、過去の事実だけを聞く。「こういう製品があったら使いますか?」は誘導であり、返ってくる「はい」に証拠能力はない。聞くべきは「最後にその問題が起きたのはいつか」「そのとき何をしたか」「いくら・何時間かかったか」。未来の意向ではなく過去の行動を掘る
- 成功基準の例: 10人中7人以上が、直近3ヶ月以内に課題の発生と具体的な対処コストを語れる

### プレセールス / LOI(発注意向書)
- 検証できる仮説: 支払意思、価格水準、購買プロセス
- 期間/費用: 2〜6週間 / 営業工数のみ
- やり方の要点: 完成前の製品を、具体的な価格・納期を提示して売りに行く。B2Bなら条件付き発注書やLOI、B2Cなら前払い・デポジット。断られた理由こそ最重要データとして記録する
- 成功基準の例: 提案10社中2社以上から、金額の入ったLOIまたは有償PoC合意を獲得

### フェイクドア / ランディングページ
- 検証できる仮説: 需要の存在、訴求メッセージ、獲得単価の当たり
- 期間/費用: 1〜2週間 / 少額の広告費
- やり方の要点: 製品説明と申込ボタンだけのページを作り、少額広告で流入させ、クリック率・登録率を測る。B2Cや低単価SaaS向き。高単価B2Bでは機能しにくい(購買が広告経由で始まらないため)
- 成功基準の例: LP訪問者の◯%がメール登録(業界の相場と比較して設定)

### コンシェルジュMVP(手動サービス版)
- 検証できる仮説: 提供価値そのもの、顧客の利用継続、価格
- 期間/費用: 2〜8週間 / 人件費のみ(開発ゼロ)
- やり方の要点: 製品が自動でやる予定のことを、まず人力で数社に提供する。価値が本物か、どの部分が本当に喜ばれるかが開発前に分かる。LLMアプリの検証と特に相性が良い(内部処理を人+汎用ツールで代替しやすい)
- 成功基準の例: 3社中2社が2ヶ月目も継続利用し、うち1社が有償化に合意

### Wizard of Oz(裏側人力の製品体験)
- 検証できる仮説: UX成立性、自動化価値、利用頻度
- 期間/費用: 2〜6週間 / 簡易フロント開発+人件費
- やり方の要点: 顧客には製品として見せ、裏側を人力で回す。コンシェルジュMVPとの違いは顧客体験が製品形式である点。利用ログが取れるため、想定利用頻度の検証に向く

### モック+ステークホルダーインタビュー
- 検証できる仮説: アウトプットの価値認知、意思決定への接続、購買関係者の反応
- 期間/費用: 2〜4週間 / モック作成工数
- やり方の要点: 実データまたは現実的なサンプルデータで「完成品が出すはずのアウトプット」を作り、利用者と購買決裁者の両方に見せる。B2Bの分析・提案系プロダクトに最適。必ず金額を聞く(「月◯万円なら稟議を出しますか」)。利用者の好評と決裁者の沈黙のギャップこそが最大の発見になる
- 成功基準の例: 決裁権者3人中1人以上が具体的な金額と予算科目を口にする

### 有償PoCの設計
- 検証できる仮説: 本契約への転換、実運用での価値
- 期間/費用: 1〜3ヶ月 / 顧客が費用負担
- やり方の要点: 無償PoCはPoC死(failure-patterns.md #4)の入口。少額でも必ず有償とし、開始前に「成功基準」と「成功した場合の本契約条件」を書面で合意する。この合意を渋る顧客は、本契約でも渋る
- 成功基準の例: PoC完了企業の50%以上が本契約へ転換

### データ検証(社内データ・公開データでの裏取り)
- 検証できる仮説: 市場規模、頻度の仮定、単位経済の変数
- 期間/費用: 数日〜2週間 / ほぼゼロ
- やり方の要点: 単位経済の表に置いた仮定値のうち、既存データで検証可能なもの(処理コスト、対象件数、現在の作業時間)を先に実測に置き換える。仮定の数を減らすだけで審査の解像度が上がる

## 提案時の書式

各検証は「仮説(何が真なら前進するか)/ 手法 / 期間 / 成功基準(数値)/ 不成立時の解釈」をセットで書く。特に「不成立時の解釈」(撤退か、ピボットか、仮説の再設計か)を事前に決めさせることで、検証結果の都合の良い解釈を防ぐ。
