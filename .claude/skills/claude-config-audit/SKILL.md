# Claude Code 設定群の棚卸しと是正

設定群は運用の中で継ぎ足されて成長する。その過程で、同じ知識が複数箇所に蓄積し、指示同士が矛盾し、古い情報や使われない設定が残り続けて、設定群の信頼性とモデルの出力品質を下げる。このスキルは設定群を棚卸しする: 継ぎ足された情報の重複を排除し、矛盾を発見し、陳腐化を取り除く。そのために証拠付きの監査報告を作り、ユーザーが承認した項目だけを是正する。

削除・統合の最終判断は常にユーザーにある。監査の質（見落としの少なさ、証拠の正確さ）がこのスキルの価値のほぼすべてであり、是正はその副産物と考えること。

## 対象

対象パスは引数で受け取る。未指定なら `~/.claude/` 配下を対象とする（プロジェクトの `.claude/` を指定されたら読み替える）。いずれの場合も対象は次の4種別のパターンに合致するファイルで、対象パスが git リポジトリ配下ならさらに git 管理ファイルに絞る。`references/` `templates/` 配下は Markdown ファイルを行数集計と棚卸し検査の対象に含め、それ以外のファイルは参照の実在確認のみ行う（役割地図の種別別検査の主対象は SKILL.md 本文）。

検査対象の4種別:

- `CLAUDE.md`
- `rules/*.md`
- `skills/*/SKILL.md`（`references/` `templates/` を含む）
- `agents/*.md`

対象外:

- コードベース本体。このスキルが扱うのは設定群の整理のみ
- `settings.json` の permissions と hooks スクリプトの中身（設定群からの参照が実在するかの確認のみ行い、変更しない）
- 依頼範囲を超えた追加整理。新しい抽象レイヤーの導入やスキルの新設・大規模な再編成は、必要だと考えた場合も報告書に提案として書くにとどめ、この実行内では行わない

## 全体フロー

次の順序を守る。Phase 3 の報告と Phase 4 の承認を経ずにファイルを変更しない（Phase 1〜3 は読み取りと作業ファイルへの記録のみ）。

1. インベントリ
2. 監査（読み取りのみ）
3. 証拠付き監査報告
4. ユーザー承認
5. 是正実行（承認された項目のみ）
6. 検証と完了確認

監査報告・承認記録などの作業ファイルは、対象パス直下の `.workspace/` に、名前に `claude-config-audit` を含む作業サブディレクトリを作って保存する（例: 対象が `~/.claude/` なら `~/.claude/.workspace/NN_claude-config-audit/`）。過去の実行記録は同じ場所から `*claude-config-audit*/approvals.md` を検索して参照し、見つからなければ初回実行として扱う。環境に別の作業ディレクトリ規約があっても、実行をまたぐ参照を成立させるため承認記録の置き場所はここに固定する。

## 棚卸し検査（主軸）

検査の中心は、継ぎ足しで蓄積した情報の棚卸しにある。核になるのは重複の排除と矛盾の発見で、これに競合・陳腐化・死蔵を加えた5つを主軸の観点として全種別を横断して検査する。以下「正典」とは、常時ロードされて挙動の基準になるファイル群（CLAUDE.md と、常時ロードされる rules）を指す。

主軸の観点:

- 重複・冗長: ファイル間での同一知識の多重管理（片方の更新でもう片方と乖離する構造）、および同一ファイル内の二度書き・再掲。指摘には一本化先の案を添える（決め方は「ファイル種別の役割地図」を使う）
- 矛盾: 両立しない指示。正典とスキル群の前提のずれを重点的に突合する。指摘にはどちらを正とするかの選択肢を添える
- 競合: 複数の設定が同じ状況を取り合う構造。skills の description が同じ要求に重なる、rules の paths glob が同じファイルに重なる、など。どれが効くかが不定になる
- 陳腐化: 廃止されたツール、古いモデル名、実体のない参照、変わってしまった機能仕様への言及
- 死蔵: どこからも参照されず、起動経路も実質ない設定。台帳に載らず、description ではトリガーされず、ポインタからも指されていないもの（`references/` `templates/` に置かれているが SKILL.md 本文から参照されない補助ファイルの孤児を含む）

補助の観点:

- 自己違反: 設定群自身が、自ら定めた規約を破っているもの。悪い例としての実演、コードフェンス内、除去対象の例示は誤検知として除外する
- 過剰規定: 読み手モデルの既定挙動で足りる手順書。判定基準は「実行時点の現行世代のモデルは、目的を1文で伝えれば妥当な手順を自分で構成できる」。内部推論を応答テキストで説明・再現させる指示もここに含める（モデルによっては refusal を誘発しうる）
- 秘匿情報の混入: API キー・トークン・秘密鍵などが設定群に紛れていないか。発見したら Phase 3 の報告を待たず即時ユーザーへ知らせる（ファイルの変更はしない）

## ファイル種別の役割地図

棚卸しの判断 — 重複をどこへ一本化するか、矛盾のどちらを正とするか、情報が本来どこに置かれるべきか — には、各種別の役割とロード特性の理解が要る。役割定義は Claude Code 公式ドキュメントの仕様に基づく: [memory](https://code.claude.com/docs/en/memory)（rules/ の仕様もここに含まれる）、[skills](https://code.claude.com/docs/en/skills)、[sub-agents](https://code.claude.com/docs/en/sub-agents)。本節の数値（行数目安・字数上限）とロード挙動の記述はこれら公式ドキュメント由来であり、実行時はリンク先の現行仕様を正とする。仕様が変わった可能性を感じたら確認して読み替える。

役割からの逸脱そのもの（サイズ超過、description の精度、frontmatter の不整合など）も、従属的な検査観点として各種別に挙げる。

### CLAUDE.md（メモリ）

役割: セッション開始時に常時ロードされ、全てのやり取りがこのファイルのトークンを背負う。公式推奨は200行以下。矛盾する指示が同居すると、モデルはどちらかを任意に選ぶ。

検査基準:

- 台帳の整合: CLAUDE.md には Rules の一覧表のみを置く方針。Rules 表は実体と突合する（掲載漏れ、存在しないエントリ、説明文と実体の食い違い）。Skills は CLAUDE.md 側に台帳を置かない方針で、`~/.claude/skills/` の実体を正典とする（台帳不在を再指摘しない）。他の種別で一覧表が存在しない場合は「要確認」として報告し、設けるか否かはユーザーが決める
- サイズ: 200行目安の超過。常時ロードする価値のない複数ステップ手順や詳細説明の居座り（公式は skill や path-scoped rule への移動を推奨している）
- 指示間の矛盾

### rules/

役割: frontmatter に `paths` がないルールは起動時にロードされ CLAUDE.md と同格に常駐する。`paths` があるルールは該当パターンのファイルを扱うときにロードされる。内容は正典（制約・キャリブレーション）か、スキルへ誘導する短いポインタ。

検査基準:

- ロード経路の整合: CLAUDE.md からの `@` import と「paths なしの自動ロード」が二重読み込みになっていないか。paths の glob が意図した対象を捉えているか
- 内容の適格性: 制約・キャリブレーションに限られているか。手順書（足場）が常時ロードに居座っていないか（スキルへ移す）
- ルール間の相互矛盾
- ポインタの健全性: 指し先が実在し、指し先に約束された内容が実際にあるか
- frontmatter 形式がルール間で統一されているか

### skills/

役割: frontmatter の description（`when_to_use` などの補助フィールドと合算で 1,536 字に切り詰め）だけが常駐し、本文はトリガー時にのみロードされる。公式推奨は本文500行以下。呼び出し名はディレクトリ名であり、frontmatter の `name` はコマンド名に影響しない。

検査基準:

- description: トリガー判定に足る具体性があるか。字数上限に収まっているか（Skills は CLAUDE.md に台帳を持たない方針のため、台帳との突合検査は行わない）。スキル間でトリガー語彙・記述言語の粒度が揃っているかを横断確認する（同種の発火条件が片方は網羅列挙・片方は一語、片方は日本語・片方は英語、のような不揃い）
- サイズと progressive disclosure: 500行目安の超過。大きな資料・テンプレート・網羅列挙は `references/` や `templates/` へ外出しする。実行時にコマンドで取得できる情報（`--help` の出力など）を本文に丸写ししていないか
- 常時ロードされるルールの再掲がないか。常駐しているルールは参照の一文すら不要
- スキル間の役割・知識の重複（棚卸し検査「重複・冗長」をスキル間に適用する）
- 手順の列挙は、結果が仕様として固定されている場合（外部規格の出力フォーマット、固定されたツール連携順序）に限られているか
- 要約や Key Principles の類のセクションが本文の再掲になっていないか
- frontmatter が用途と整合しているか:
  - `allowed-tools` と本文要求の突き合わせ。不足（本文の手順や Current state 記法（感嘆符 + バッククォート囲みコマンド）が要求する操作の未宣言 = 実行時権限プロンプトの原因）/ 過剰（本文に使い道のない宣言）/ 書式（Bash は `Bash(cmd:*)` のコロン式、非 Bash ツールは素の形式、エントリのスペース区切りは排除）
  - `disable-model-invocation` と description の整合。自動発火を促す description の文面と手動専用フラグ（`disable-model-invocation: true`）の矛盾を検出する

### agents/

役割: 独立したコンテキストで動くサブエージェント定義。登録名は frontmatter の `name` でありファイル名は無関係。`model` 省略時は inherit（セッションのモデルを継承）、`tools` 省略時は全ツール継承。

検査基準:

- name の一意性（同一 scope 内で重複すると片方しかロードされない）
- description が委譲判定に足る具体性を持つか
- `tools` / `model` の省略が意図的か。本文で特定ツールの使用を宣言しながら tools 制約が未設定（不足）、逆に本文の設計意図より広い権限を保持（過剰。例: 助言専用エージェントが Write/Edit/Bash を保持）、といった不整合がないか
- ファイル名と `name` の不一致は仕様違反ではないが、紛らわしさとして「要確認」で報告する
- skills との役割重複（同じ用途のスキルとエージェントが併存し、使い分け基準がない）
- 委譲・参照先が実在し、参照先に期待した内容があるか

## 判断基準（指摘の分類）

すべての指摘に次のいずれかの分類を与える。分類が推奨の起点になる。制約・キャリブレーション・足場の3つは「この指示を残すか」を問う指摘に、事実整合は正誤が客観的に決まる指摘に使う。

| 分類 | 定義 | 既定の扱い |
| --- | --- | --- |
| 制約 | 安全・業務上の絶対条件。破ると実害が出る | 保持。表現の簡潔化のみ可。正典1箇所に集約 |
| キャリブレーション | 好み・スタイルの調整。モデルの既定挙動を特定方向に寄せる | 保持しつつ最短の表現に簡潔化 |
| 足場 | 旧世代・下位モデルの失敗を防ぐための手順書・レシピ | 読み手モデルの既定挙動で足りるなら削除候補。迷ったら意図1文への圧縮を推奨 |
| 事実整合 | 正誤が客観的に決まる不整合（実体のない参照、台帳のずれ、重複、陳腐化した事実） | 書き換え・統合を推奨。どちらを正とするかの決定が要るものは承認で個別に問う |

「要確認」は分類・推奨のどちらにも使える値で、ユーザーの判断なしに先へ進められない指摘（分類に迷う、不在の一覧を設けるか等の意思決定を含む）を示す。要確認の指摘は推奨を保留し、勝手に削除へ倒さない。

## Phase 1: インベントリ

- 対象ファイル一覧と各ファイルの行数、種別ごとの合計、総行数を集計する
- CLAUDE.md に一覧表が存在すれば実体と突合する（扱いの詳細は「ファイル種別の役割地図」の CLAUDE.md の項に従う）
- 結果をベースラインとして `baseline.md` に記録する。Phase 6 の増減測定の基準になる

## Phase 2: 監査

- トピック単位の横串突合を主手順とする: 同じ主題の知識が全種別のどこに散在するかを、主題ごとに揃えて読む（起点の例: 正典が扱う各主題、台帳の各行）。重複・矛盾・競合はここで見つかる。読み始めは常時ロードされる正典から
- 続いて役割地図の種別別検査で各ファイルを確認する。参照・起動経路の突合による陳腐化・死蔵の検出と、補助の観点（自己違反・過剰規定・秘匿情報）もこのパスで拾う
- 規模が大きければサブエージェントに観点・種別単位で分担させてよい（引用の検証責任は Phase 3 の報告の規律に従う）
- grep による定量補助を使う（強調記号や強制語の頻度、常駐ルール名の再掲検索、行数集計など）

## Phase 3: 監査報告

報告書を `audit-report.md` に書き、同じ内容をユーザーに提示する。各指摘は次の形式（`LL` は行番号、`NN` は数値のプレースホルダ）:

```markdown
### R-NNN: <1行サマリー>

- 位置: `path/to/file.md:LL-LL`（矛盾は両側を併記）
- 引用: 実際の文面（1〜2行）
- カテゴリ: 重複・冗長 / 矛盾 / 競合 / 陳腐化 / 死蔵 / 役割逸脱 / 自己違反 / 過剰規定 / 秘匿情報
- 分類: 制約 / キャリブレーション / 足場 / 事実整合 / 要確認
- 推奨: 削除 / 簡潔化 / 統合 / 書き換え / 保持 / 要確認
- 理由: 1〜2文
- 削減見込み: 約NN行（分かる場合）
```

報告の規律:

- 引用は、サブエージェントの報告由来であっても、自分の Read / Grep の結果と一致していることを確認してから載せる
- 証拠（位置と引用）を示せない主張は報告に含めない。不在を主張する指摘（死蔵・台帳の掲載漏れなど）は、文面の代わりに探索の結果（grep コマンドと0件の出力、突合表など）を証拠として添える
- 推測は「要確認」と明記して事実と区別する

末尾にサマリーを付ける: 指摘総数、カテゴリ別件数、削減見込み合計、対応の優先順位案。

## Phase 4: ユーザー承認

- AskUserQuestion で承認を取る。性質の同じ指摘はグループ化して問うてよい。ただし矛盾の解消のように「どちらの系統を正とするか」の意思決定を含むものは必ず個別に問う
- 総行数の削減目標もここで合意する。目安は承認項目の削減見込み合計。台帳の補完や矛盾解消の書き換えなど行数が増える正当な是正もあるため、目標は努力目標として扱い、完了の必須条件にはしない
- 結果を `approvals.md` に記録する: 指摘ID、判断（承認 / 却下 / 保留）、ユーザーの指定事項。却下された指摘は理由とともに残し、次回の監査で同じ指摘を再生産しないための参照にする
- 承認されなかった項目は実行しない。保留は次回監査に持ち越す
- 却下・保留の項目は `TaskCreate` に登録しない。Phase 5 のタスクリストには承認済み R-NNN のみが乗る（タスクリスト上の `deleted` は「承認済みだが実施しなかった」ケース専用として使い、却下と機能的に分離する）
- 承認された項目が0件なら Phase 5 と Phase 6 の検証は行わず、監査報告と記録（`approvals.md` への判断の記録）を成果として完了する
- 承認された項目が1件以上あれば、Phase 5 へ進む前に承認済み R-NNN を `TaskCreate` で登録する。content は `R-NNN: <指摘の1行サマリー>` の形式で R-NNN と対応づける。「1 指摘（承認済み R-NNN）= 1 タスク」を登録単位とし、承認が1件だけでもタスクリストを作る（Phase 5 の進捗と Phase 6 の完了確認を機械化するため。粒度宣言と最小差分編集哲学との一致は Phase 5 追記参照）

## Phase 5: 是正実行

承認済み項目のみ、1指摘 = 1編集の最小差分で実施する。

書き換えの原則:

- 意図を1文で言えるなら手順を捨てて意図を書く。理由（why）は残し、手順（how）を削る
- 制約は正典の1箇所に集約し、他所からは削除する
- 矛盾の解消は、承認時に決めた正の系統へ全ファイルを揃える
- ファイル移動・改名は、対象が git 管理下なら `git mv` を使う
- 台帳と実体を一致させる
- コミットはユーザーから指示があるまでしない

進捗報告の規律: ユーザーに進捗や完了を報告する前に、報告しようとしている各項目を、このセッションで実際に実行したツール結果（Edit の成功、変更後の該当行）と照合する。照合できた作業だけを完了として報告し、未実施・失敗・スキップはその旨をそのまま書く。

## Phase 6: 検証と完了確認

1. fresh context の検証: 新しいサブエージェントに書き換え後の設定群を読ませ、「矛盾している箇所・冗長な箇所・意味の取れない箇所」を指摘させる。このとき監査報告や会話の文脈を渡さない（先入観のない読者として読ませるため）。規模が大きければ Phase 2 と同様に分担させてよい
2. 検証者の指摘は `approvals.md` の却下記録と突合し、却下済みと同内容のものを除外する。残った矛盾の指摘は Phase 3 の形式で整理し、Phase 4 の承認から再度回す。矛盾以外の新規指摘（冗長・不明瞭）はユーザーに提示のうえ次回監査へ持ち越してよい。再ループの必須条件は矛盾の解消のみとする
3. 完了条件（すべて満たしたことを確認してから完了を宣言する）:
   - 検証サブエージェントからの矛盾の指摘（却下済みと同内容のものを除く）が0件
   - 承認済みの全項目が実施済み、またはスキップ理由が `approvals.md` に記録済み
   - すべての変更が `approvals.md` のいずれかの承認と対応している（承認のない変更が diff に存在しない）
   - 総行数の増減がベースラインと対比して記録され、Phase 4 の削減目標に届かない場合は理由が最終報告で説明されている
   - 台帳と実体が一致している（ユーザーが却下した項目に起因する不一致を除く）
   - Markdown linter のエラーが未解消のまま残っていない（hook 未設定の環境では利用可能な linter があれば実行し、なければこの項は対象外としてよい）
4. 最終報告: ベースラインとの対比（ファイル数・総行数）、実施 / 却下 / 保留の件数、次回監査への持ち越し事項

## 定期運用

このスキルは一度きりではなく、スキルを数個継ぎ足すたび、またはモデル世代が変わるたびに回す。`approvals.md` の保留・却下記録を次回監査の入力にする。
