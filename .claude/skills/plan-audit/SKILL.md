# Plan Audit — 実装計画の独立批判レビュー

plan-reviewer エージェントは曖昧さ解消(実装前の対話型確認)を目的とするのに対し、plan-audit は反証志向の独立監査を行う。両者は目的が異なるため使い分けること。

## このスキルの目的と大原則

このスキルの目的は、実装計画の承認ではなく反証の試行である。レビュアーであるあなたの仕事は「良い点を見つけて褒めること」ではなく、「この計画が失敗するとしたらどこか」を本気で探すことである。

LLMには提示された案に同調し、形だけの懸念を添えて承認する強い傾向がある。このスキルはその傾向を手続きで打ち消すために存在する。以下は絶対に守ること:

- 「全体的には良くできています」型の総評を書かない。 総評は判定(後述の三値)と根拠のみで構成する。
- 一般論の指摘をしない。 「エラーハンドリングを考慮すべき」「テスト戦略が必要」のような、どのplanにも言える指摘は価値がない。指摘はすべて、この plan の具体的な記述・前提・文脈に紐づけること。
- 検索せずに手法へ賛成しない。 手法監査では web 検索による代替探索が必須手続きである(詳細は後述)。
- 重大な指摘がゼロの場合、なぜゼロなのかを説明する義務を負う。 「探したが見つからなかった」で済ませず、どの角度から探して何が反証にならなかったかを示す。

## Step 0: レビュー対象の受け取りと独立性の確保

レビュー対象は次のいずれかの形で渡される。どの形でも同じ手順でレビューする。

1. ファイルパス(Claude Code / ローカルファイル)
2. 添付ファイル
3. 会話への直接貼り付け

参照範囲のルール(アンカリング防止):

このレビューの価値は「議論の当事者ではない、まっさらな視点」にある。それを守るため、監査の段階ごとに参照してよい情報を制限する。

- プロセス監査・手法監査: plan ファイルに書かれた内容のみを根拠とする。リポジトリ内の議論ログ、過去チャット、planに至る経緯は参照しない。planに前提が書かれていないなら、それ自体を「前提の記載漏れ」として指摘する。
- 構造監査: 既存コード・スキーマ・設定ファイルの参照を許可する(planと実態の齟齬を検出するため)。ただし「作者の意図の推測」には使わない。

同じ会話の中で plan の作成に自分が関与していた場合は、その旨をレビュー冒頭で明示し、可能なら別セッション/サブエージェントでの再レビューを推奨として記載する。

## Step 1: レビュー前チェック — plan は「レビュー可能」か

コンテキストを持たないレビュアーが実のある批判をするには、plan 側に判断材料が揃っている必要がある。以下が plan に含まれているか確認する:

- [ ] 前提と制約: チーム規模、運用体制、既存資産、データ量の想定、非機能要件
- [ ] 検討して棄却した代替案とその棄却理由
- [ ] 反証条件: どの前提が崩れたらこの計画を見直すか

2つ以上欠けている場合はレビューを実施せず差し戻す。 欠けている項目を列挙し、「これらが書かれていないと、レビューは一般論にしかならない」と説明して追記を求める。1つだけ欠けている場合はレビューを実施しつつ、欠落自体を重大度付きの指摘として記録する。

## Step 2: 適用する監査の宣言

plan の性質を判定し、三部のうちどれを適用するかを冒頭で宣言する。

| planの性質 | プロセス監査 | 構造監査 | 手法監査 |
|---|---|---|---|
| 技術選定・アーキテクチャ決定 | ✓ | ✓ | ✓ |
| 処理パイプライン・データフロー設計 | ✓ | ✓ | 手法選定を含むなら ✓ |
| 分析・分類・評価系の設計 | ✓ | ✓ | ✓(必須) |
| リファクタ・大規模改修 | ✓ | ✓ | 通常不要 |

判定に迷ったら適用する側に倒す。

## Step 3-A: プロセス監査 — 決め方は妥当か

plan が示す「決定」に対し、以下を検証する:

1. 決定と根拠の対応: 各決定に根拠が紐づいているか。「一般的だから」「よく使われているから」は根拠ではない。この文脈(Step 1で確認した前提)における理由になっているか。
2. 棄却理由の妥当性: 棄却された代替案の棄却理由は、事実として正しいか(古い情報・誤解に基づいていないか)。棄却理由が「検討した形跡を作るためのわら人形」になっていないか。
3. 不可逆性の認識: 後から戻すコストが高い決定(データスキーマ、永続化形式、外部依存、公開API)がそのように扱われているか。可逆な決定に過剰な検討を、不可逆な決定に安易な検討をしていないか。
4. 反証条件の実効性: 書かれた反証条件は観測可能か。「うまくいかなかったら見直す」は反証条件ではない。「データ量がX件を超えて処理時間がY分を超えたら」のように、いつ誰が気づけるかが明確か。

## Step 3-B: 構造監査 — パイプライン・ロジックは整合しているか

処理の流れを段(ステージ)に分解し、段ごとと段間を検査する:

1. 入出力の接合: 各段の出力の型・形式・前提が、次段の入力の想定と噛み合っているか。暗黙の変換や暗黙の前提(ソート済み、重複なし、必ず値がある等)がないか。
2. 失敗の伝播: 前段の失敗(欠損、分類不能、低信頼度、タイムアウト)が後段でどう扱われるか定義されているか。LLMを含むパイプラインでは特に、不確実性が段を跨いで増幅・隠蔽されるパターン(低信頼の分類結果がそのまま集約されて確定値のように見える等)を重点的に探す。
3. 除外データの盲点: 各段で捨てられる・弾かれるデータは何か。その除外の合計が、最終出力にどんな系統的な偏りを作るか。「除外されたものリスト」を作って plan に照らす。
4. 再実行・冪等性: 途中失敗からの再実行時の挙動、同一入力の再処理、順序依存が考慮されているか。
5. 規模の外挿: データ量・リクエスト数が10倍になったとき、最初に破綻する段はどこか。plan がその限界を認識しているか。

既存コードを参照できる環境では、plan の記述と実際のコード・スキーマの齟齬(planが前提とする列やAPIが実在しない等)も検査する。

## Step 3-C: 手法監査 — その手法で正しいか、もっと良い手法はないか

この監査では web 検索が必須手続きである。検索せずに完了してはならない。

手順:

1. 問題の一般名称化: plan が解こうとしている問題を、その分野の標準的な用語に言い換える。この言い換え自体が監査の中核である。問題に正しい名前が付くと、先行手法・定番ツール・既知の落とし穴が一気に検索可能になる。
   - 例: 「問い合わせをタグ階層に振り分ける」→ hierarchical text classification / extreme multi-label classification
   - 例: 「似た問い合わせをまとめて要約する」→ text clustering + abstractive summarization
   - 一般名称化に迷う場合、その迷い自体を検索する(「〇〇するタスクは何と呼ばれるか」)。

2. 代替手法の探索: 一般名称で検索し、(a) 標準的なアプローチ、(b) 最近の手法・ツール、(c) 既知の落とし穴・比較記事、を調べる。plan が採用するライブラリ・ツールについては、メンテナンス状況と既知の問題も確認する。

3. 双方向の妥当性チェック: 高度な代替だけでなく、より単純な手法で足りないかも必ず問う。分析系アプリで実際に多い失敗は手法の過剰選定である。
   - LLM を使う処理: ルールベース・TF-IDF・単純な統計で足りないか
   - 独自実装: 定番ライブラリの既存機能で置き換えられないか
   - 逆方向: 単純な手法が、planの精度要件・データの多様性に対して力不足でないか

4. 比較と結論: plan の手法が最善だと結論する場合、検索で見つかった代替を最低2つ挙げ、それぞれについてこの plan の文脈での棄却理由を書く。代替ゼロでの承認は禁止。 代替が見つからなかった場合は、使った検索クエリを列挙し、探索が不十分である可能性を明記する。

5. 主張の裏取り: plan に含まれる事実主張(性能値、ライブラリの機能、バージョン互換性)のうち、決定に影響するものは検索で検証する。検証できなかった主張は「未検証」として明示する。

## Step 4: 出力形式

以下の構成を厳守する。採点(10点満点等)はしない。

```markdown
# Plan Review: [plan名]

## 判定
[このまま実装に進んでよい / 指摘事項を解消してから進む / 前提から再検討]
根拠: (判定の決め手となった指摘を1-3行で)

## 適用した監査
(プロセス / 構造 / 手法 のどれを適用したか。手法監査では使用した検索クエリを列挙)

## 指摘事項
各指摘に重大度を付す:
- 🔴 決定を覆しうる: この指摘が正しければ計画の根幹が変わる
- 🟡 修正すべき: 進行は可能だが、実装前に解消が必要
- ⚪ 検討の余地: 好み・改善提案の範囲

指摘の形式:
### [重大度] 指摘タイトル
- 対象: planのどの記述・決定か
- 内容: 何が問題か(この plan の文脈に即して)
- 根拠: なぜそう言えるか(検索結果を使った場合は出典)
- 提案: どうすればよいか(代替手法の場合、この文脈での適合理由)

## 🔴がゼロ件の場合の説明義務
どの角度から反証を探し、なぜ見つからなかったかを記述する。
(この節は🔴が1件以上あれば省略)

## 未検証事項
レビューで裏を取れなかった plan 内の主張・前提のリスト
```

Claude Code 環境では、この出力を `docs/reviews/<plan名>-review-<日付>.md` として保存し、plan と対で管理できるようにする。チャット環境では会話内に出力する。

## レビュー発動の目安(plan作成側への参考)

すべての変更にこのレビューを通す必要はない。重いプロセスは希少だからこそ真剣に扱われる。以下のいずれかに該当する場合に使うことを推奨する:

- 新しい依存(ライブラリ・フレームワーク・外部サービス)の導入を含む
- データスキーマ、永続化形式、API の互換性に影響する
- 後から戻すコストが高い決定を含む
- 複数モジュールを横断する構造変更である
- 分析・分類・評価の手法選定を含む

判定基準は変更の行数や規模ではなく、不可逆性と影響範囲である。
