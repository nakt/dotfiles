# プラン作成ルール

プランを作成・更新した後は、必ず以下を実行すること:

1. plan-reviewer サブエージェントを呼び出す
2. サブエージェントが曖昧な点を指摘した場合、ユーザーの回答を待つ
3. 回答を受けてプランを更新
4. 全ての曖昧さが解消されてから実装開始

## プランの構成: 承認用セクション

プランは「前半 = 承認者（人間）向け / 後半 = 実装者向け」の二部構成とする。承認者がタイトル直下だけを読めば承認判断ができるように、以下 5 セクションをこの順序・見出し名（正規名）で置く。

1. `## 要約`（全プラン必須）: 5〜10 行を目安に「何を・なぜ・どう変えるか」を書く。プラン固有の略語・コードネーム・内部フェーズ名を使わない。「ここだけ読めば承認判断ができる」ことが基準
2. `## 挙動の Before/After`（観測可能な変化があるとき必須）: コマンド・出力・運用手順など、ユーザーから観測できる変化のみを表にする。内部実装の変化は書かない
3. `## この計画で確定する判断`（代替案を比較して選んだ判断があるとき必須）: D1, D2… と番号付きで各 1〜2 行にまとめ、採らなかった代替案を 1 行添える。本文に承認対象の判断を書いたら必ずこの一覧にも反映する（散在禁止）
4. `## 用語表`（プラン固有の略語があるとき必須）: プラン内で 2 回以上使う略語・コードネーム・内部名称を定義する
5. `## AskUserQuestion 合意事項`（AskUserQuestion で確定した項目があるとき必須）: Q1, Q2… と番号付きで「質問要旨: <採用した回答> — 判断根拠: <1 行>」の形にまとめる。D 番号との重複は可（同じ判断が「代替案から選んだ」でも「AskUserQuestion で確定した」でもある場合、両方に載せる）。execute-plan の controller はここを implementer / reviewer prompt に転記するため、プラン合意事項をここに集約しないと引き渡されず、生成コードが合意と食い違う原因になる

承認用セクションはプランの冒頭（本文セクション `## Context` 等や後半の `## 実装タスク` より前）に、上記 1〜5 の順序で置く。該当がないセクションは見出しごと省略してよい（空セクションを残さない）。存在するセクションについては順序と見出し名を変更しない（将来の機械抽出の余地を残す）。

`## AskUserQuestion 合意事項` の記述例:

```markdown
- Q1. 質問要旨: 認証方式の採用 — 採用回答: OAuth 2.0 (Google) — 判断根拠: 既存 SSO 資産と整合、実装コスト低
- Q2. 質問要旨: 対象言語スコープ — 採用回答: Python + TypeScript の 2 系統 — 判断根拠: 実案件で使う 2 大言語をカバー、他言語は次回摩擦発生時に追加
```
