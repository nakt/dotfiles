# サブエージェント委譲ルール

サブエージェントにタスクを委譲するとき、どこまで文脈を切って渡すかの指針。闇雲に全部を並列サブエージェントに散らすと、質の低い結論を量産して「浪費」に終わる。切ることの効用と害は同じ操作から出るので、タスクの性質で切り分ける。

## トレードオフの核心

サブエージェントに文脈を切って渡すのは、親の文脈汚染を避けて客観的・独立に考えさせる正当な狙いがある。探索の方向を分散させたい、親の思い込みを持ち込ませたくない場面では有効に効く。

一方で「客観性のための文脈カット」と「判断に必要な文脈の欠落」は地続きで、線引きが難しい。客観性を期待して切ったつもりが、子から見ると単に判断材料が足りない状態になり、埋め合わせに推測で走る。独立性を狙ったはずが、質の低い独立結論を量産することになる。効用と害が同じ操作から出る、という前提を忘れない。

## 探索寄りか判断寄りかで切り分ける

委譲するタスクが探索寄りか判断寄りかを見極める。

- 探索寄り（候補を広げる / 可能性を列挙する / 別の角度から当たる / 調査する）は、文脈を薄くしても壊れにくい。むしろ薄いほうが親の枠に縛られず広がる。積極的に散らしてよい。
- 判断寄り（何が正しいか決める / 優先順位をつける / 要件に照らして採否を出す）は、判断基準そのものが文脈の中にある。切ると基準ごと失われ、子は自前の適当な物差しをでっち上げる。

## アンチパターン: 探索のつもりで判断を投げる

トンチンカンな結論が出やすいのは、探索のつもりで判断を投げているとき。候補列挙のつもりで採否まで委ねる、調査のつもりで優先順位づけまで委ねる、といった混同を避ける。判断を伴うなら、探索と同じ軽さで散らさない。

## 切るなら物差しは残す

判断寄りのタスクをどうしても委譲するなら、タスクそのものより「判断基準」と「何を根拠と認めるか」は残す。生の親コンテキストは渡さず客観性は保ちつつ、評価の物差しだけ共有する。

物差しがないまま客観性だけ求めると、各自が別の物差しで測った結論が返ってきて、統合段で親がまた揉めることになる。物差しを渡せば、独立性を保ったまま噛み合った結論が集まる。

## 運用粒度

- 探索は散らす。
- 判断は親が握る。委譲するなら基準を明示した上で少数に絞る。

## 実装例

このルールを具体化したワークフローが `.claude/skills/execute-plan/` にある。controller（親）がコンテキストを curate し、実装は fresh subagent に散らしつつ、設計採否などの判断は controller / ユーザーへエスカレーションさせ、reviewer には評価の物差し（Acceptance criteria とレビュー観点）を必ず渡す構造になっている。
